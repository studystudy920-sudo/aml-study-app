# 不正検知・インシデント対応プレイブック（7つの打ち手）

不正取引検知時に個人IB／法人IBがどう動くかを、2回の会議から「7つの打ち手」として整理したもの。

- **第1回**: 情報セキュリティ部門（情シス／CSIRT）へのヒアリング（サイバー／システム視点）
- **第2回**: 業務部門（個人デジタルビジネス推進部＝RBD／AML）との協議（業務・初動視点）。第1回の理解を実務面から補正。

本アプリの「クイズ」タブに同名の学習モジュール（`不正検知・インシデント対応`／7章・42問）として収録している。

## 前提・全体像

- **不正取引検知への対応は「サイバーセキュリティ事案」として扱う。** サイバー／システム面の統括は情報システム部門／CSIRT。
- 手順の骨格は **サイバーセキュリティ管理マニュアルのプロセスマップ**（現状は大枠のみ。レベル判定・個別対応の細部は未整備＝要具体化）。
- 連絡先・代替手段など業務側の初動は **各業務のコンティンジェンシーマニュアル**（社内イントラの規定マニュアル＝当行全規則、オペレーション統括部の規則配下。スタビビ／個人IB／法人IB別）。
- IBは委託先ベンダーへの **共同利用（アウトソース）**。ベンダーがIDS/IPS含め24/365で監視・検知するが、**当行としての障害判定・判断はIT部門が行う**（後述の重要な補正点）。
- **核心の違い**: 脆弱性対応等は **全停止** 前提、**不正検知は一部停止** が基本。「何を・誰が止められるか」を事前に定めるのが本件のお題。

## 第2回会議での重要な補正点（第1回からの見直し）

1. **検知・障害判定はIT部門が起点**。ベンダー（NRI等）が検知しても、当行としての判断はIT部門。基盤運用の警報→チャネル連携→（ベンダー起因なら）ベンダー連絡＋行内連絡という経路。
   - 業務部門（RBD）はベンダーから「顧客問い合わせが多いので確認を」と連絡を受けるが、ほぼ100%「障害です」とは言われない。**「ベンダー→業務部門が起点で障害宣言」という理解は誤り。** 情シス側にも認識のズレがあり、要すり合わせ。
2. **対策本部を立ち上げるのは業務BUではない**。危機対策本部＝**リスク統括部**が設置／システム対策本部＝**IT部門**が設置。業務部門は対策本部の指示のもと、規程上の役割（Web掲載・告知、代替策指示、緊急連絡網）を担う＝比較的後段。
3. **対策本部の前に「現場の緊急会議体」がある**。検知直後は各部が断片情報しか持たないため、関係現場（AML・コンプラ・IT・RBD等）が集まり情報集約＋役割分担する緊急会議が実質的な初動。
4. **レベル判定が必要**。システム対策本部（IT）は障害レベル判定を行い、度合いによっては対策本部を立ち上げず通常復旧で済む。不正も同様の「不正レベル判定」を設けて具体化する（現状は概要のみ）。

## 7つの打ち手

### ① 初動・検知受付・対応記録
- **実際の初動は現場の緊急会議体**：関係部署が集まり情報集約・役割分担（2020年前例）。
- **顧客受付窓口は業務部門（AML／RBD、コールセンター）**。ここに情報を集約し、どこへ発信・ITへ調査依頼するかを判断。
- **記録・会議体（Teams/WebEx）設営・ベンダー接続などの事務局機能はCSIRT等が担う。** 業務部門に記録まで負わせると初動が遅れる。一人では書ききれないため分担して残す運用が重要（CSIRT主管、デルタウォール等の訓練で練習）。
- **論点**: 予防的停止時の事前招集・記録の要否は未整理のグレーゾーン。

### ② エスカレーション・対策本部
- **危機対策本部＝リスク統括部が設置**（大規模災害・顧客被害・情報漏洩確認時）／**システム対策本部＝IT部門が設置**（システム起因）。
- 業務BUの一部署が対策本部を立ち上げることはない。設置後にその指示のもとで役割を担う。
- **レベル判定**：度合いによっては対策本部を立ち上げず通常復旧。目安例＝不正は実損害が1件でも出たら立ち上げ検討／リスク統括の累積被害金額基準 **3,000万円**（個人＝累積、法人＝1件でも）。
- 取締役への速報等の手順はプロセスマップに準拠。不正取引検知もサイバー事案として扱う。

### ③ 影響範囲・原因の特定
- **当行としての障害判定・判断はIT部門**（開発＝チャネル系、基盤運用）。ベンダーは24/365で検知・解析。
- 所管の切り分け: **障害＝運用・ITチャネル部門／サイバー＝情シス・CSIRT／開発起因＝開発部門**。ベンダーと手分けして影響を特定。
- CSIRTは第二線として、調査観点の網羅性や漏れをレビューし記録。
- 情報収集先として金融ISAC（脅威情報共有）等も活用。

### ④ システム停止（一部停止と3段階）
- **不正検知は一部停止が基本**（全停止型の脆弱性対応と異なる）。
- チャネル停止の「3段階」＝①ホームページ（入口）②ベンダーのAPI ③後段の勘定系接続。ただし**初段に何を止めるかは事象による**（例：アプリのログインAPIだけ止め個人IBへ誘導／特定連携APIだけ止める／ページ入口封鎖）。
- 停止範囲は障害・被害の条件でジャッジ（個人IB・法人IBとも概ね同様）。

### ⑤ 口座ロック・取引制限
- 不正送金は全体停止より **被害口座など利用者アカウント（ID）の個別ロックが基本**。
- **実行手段＝運用管理端末（運端）**：振込停止・送金機能停止が可能。実施はコールセンター・事務（カスタマーサービス）で、権限は管理者権限・CC権威者・Web登録権威者。業務部門（RBD）が依頼。
- **金融犯罪対策（AML）**は顧客申告（被害・認証情報漏洩等）に基づき **自部門の判断で一旦停止**（コールセンター/監視センター）。金融犯罪以外は関係部署で協議のうえ依頼。個別指示は全店へのメール等。
- 影響が広ければ「送金だけ停止」等へ範囲を拡大、究極はIB全体の送金停止という大きな判断も。

### ⑥ 顧客・当局・関係先連絡
- 当局（金融庁・日銀）への窓口は **経営企画部**。重大時は経営企画から当局・親会社等へ、広報総括部も関与。
- 連絡フローは通常のシステム障害時のフローと同じ。顧客向けはホームページのトピックス等でアナウンス。
- 2020年前例では顧客連絡・警察通報を並行実施。

### ⑦ コンティンジェンシー・復旧
- 代替手段は各業務のコンティンジェンシーマニュアルに定義。**個人IB＝コールセンター/窓口、法人IB＝全店窓口/部署誘導。**
- 共同利用システムの復旧はベンダー次第で、長期戦になり得る。

## 2020年の前例（教訓）

- 不正送金1件＋ログ調査で複数被害が判明→一旦IB停止／人身停止を決定。
- 全クローズではなく、ログイン認証を変更し「合言葉登録済みの通常利用者は利用可・未登録者は不可」とした。しかし未登録者の方が多く影響が大きかった。
- 手順が明確でなく口頭でワーッと決めて止めた。承認は関係役員がチャット上で即断。→ 今回のお題は「**何を止めるか（送金だけ止める等）をきちんと**」定めること。

## 積み残し課題

- **モバイルアプリの技術的負債**: 統合開発環境のバージョンが古く（ストア規定の対応世代を外れる）、パッチ改修版をアプリストアに掲載できないおそれ。開発基盤のバージョンアップが課題。
- **レベル判定の具体化**: プロセスマップは大枠のみ。個別対応／本部設置の境界、金額・件数基準を整理。
- **記録運用**: 分担して残す体制（CSIRT主管）を訓練で確立。
- **予防的停止**時の事前招集・記録の要否を整理。
- 当局要請事項はヒアリング（アンケート）シートで各ベンダーに送付・回答収集。

## 次のステップ

- 月曜の個人（稲山さん等）会議はリスケ→**再来週**。まず今日のインプットで大枠フロー／フレームワークを作成→本メンバーで既存マニュアルとの違和感を照合→部長（羽田さん等）へ確認、という順。
- 進め方は「既存資料ではこうなっている／抜けている所をこうしたい」の頭出し＋案提示。
- 法人版は先行してたたき台を作成（法人はオペレーション統括部／法人企画の役割分担が複雑）。個人・法人とも同じ流れ＋既存コンチプランで整備し、不足分を洗い出す。
