# 不正検知・インシデント対応プレイブック（7つの打ち手）

不正取引検知時に個人IB／法人IBがどう動くかを、情報セキュリティ部門（情シス／CSIRT）へのヒアリング会議の内容から「7つの打ち手」として整理したもの。
本アプリの「クイズ」タブにも同名の学習モジュール（`不正検知・インシデント対応`／7章）として収録している。

## 前提・全体像

- **不正取引検知への対応は「サイバーセキュリティ事案」として扱う。** 主導・統括は情報システム部門／CSIRT。
- 手順の骨格は **サイバーセキュリティ管理マニュアルのプロセスマップ** に定義。エスカレーション・対策本部立ち上げ・当局連絡までの流れが載る。
- 連絡先・代替手段など業務側の初動は **各業務のコンティンジェンシーマニュアル**（社内イントラの規定マニュアル＝当行全規則、オペレーション統括部の規則配下。スタビビ／個人IB／法人IB別）に定められている。
- IBは委託先ベンダーへの **共同利用（アウトソース）**。IDS/IPSを含め24時間365日で監視・解析され、金融機関はその結果を受領して第二線的にレビューする。

## 7つの打ち手

### ① 初動・検知受付・対応記録
- 検知の一次受付は各業務のコンティンジェンシーマニュアルに定めた受付窓口。そこからCSIRTへ連携し招集。
- **記録・会議体（Teams/WebEx）設営・ベンダー接続などの事務局機能はCSIRT等の専門部門が担う。** 初動でバタバタする局面で業務部門に記録まで負わせると初動が遅れ、後続のエスカレーション全体が遅延する。
- 対応にはシステム対策本部用のTeamsチャネル（既設）を利用できる。
- **論点（グレーゾーン）**: 他行でも被害拡大時の「予防的停止」では、被害発生前の事前招集・記録の要否が未整理。あらかじめ決めておく必要がある。

### ② エスカレーション・対策本部
- CSIRTが **システム対策本部** を立ち上げ、被害の度合い（レベル4・5、金額基準）に応じて **危機対策本部** へ格上げ。
- 取締役への速報等の手順はプロセスマップに準拠。
- サイバー事案の統括は情シス／CSIRT（従来のIT部門主導から所管変更）。金額基準はマニュアルで要確認。

### ③ 影響範囲・原因の特定
- 所管の切り分け: **障害＝運用・ITチャネル部門／サイバー＝情シス・CSIRT／開発起因＝開発部門**。ベンダーと手分けして影響を特定。
- ベンダー（例: 委託先SIer）が一次解析、CSIRTは「必要な観点の調査がなされたか」「漏れはないか」を第二線でレビューし記録。

### ④ システム停止（3段階）
停止範囲は3段階で設計。障害・被害の条件に応じてどこまで止めるかをジャッジする（個人IB・法人IBとも概ね同様）。

1. ホームページ（入口）
2. ベンダーのAPI
3. 後段の勘定系接続

### ⑤ 口座ロック・取引制限
- **不正送金は全体停止よりも、被害口座など利用者アカウント（ID）の個別ロックが基本。**
- ID停止は個人IB／法人IBの運用・カスタマーサービス部門が持つ **運用管理端末** で実施。夜間はATM監視センターで一旦停止する運用もあり、過去に口座停止の実績あり。
- 根拠は不正送金（AML）対応マニュアル。フィッシング等で口座認証情報が漏れた場合はID停止。
- 個別ロックで足りるか、入口から止めるか、勘定系まで影響が及ぶかをレベル判断する。

### ⑥ 顧客・当局・関係先連絡
- **当局（金融庁・日銀）への窓口は経営企画部。** 重大時は経営企画から当局・親会社等へ連絡、広報総括部も関与。
- 連絡フローは通常のシステム障害時のフローと同じ。
- 顧客向けはホームページのトピックス等でアナウンス。緊急体制・動作確認要員の確保・顧客告知の段取りも準備。

### ⑦ コンティンジェンシー・復旧
- 代替手段は各業務のコンティンジェンシーマニュアルに定義。**個人IB＝コールセンター/窓口、法人IB＝全店窓口/部署誘導。**
- 共同利用システムの復旧はベンダー次第で、業務的に長期戦になり得る。

## 積み残し課題

- **モバイルアプリの技術的負債**: 統合開発環境のバージョンが古く（アプリストア規定の対応世代を外れる）、パッチ改修版をストアに掲載できないおそれ。開発基盤のバージョンアップが課題。
- 予防的停止時の事前招集・記録の要否（①の論点）を整理・決定する。
- 対策本部立ち上げ判断に用いる被害金額の基準をマニュアルで確認・明確化する。
- 当局要請事項はヒアリング（アンケート）シートに落として各ベンダーに送付・回答収集する運用。
